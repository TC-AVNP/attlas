## Lessons

- **Always test flags before shipping them.** The plan said
  `--title-format=attlas` for ttyd and `--app-name=attlas` for
  code-server. ttyd actually uses `-t titleFixed=...` (client
  option), and code-server's flag is `--app-name`. Ran `--help` on
  the VM for both before writing the install-script edits.
- **System-wide `safe.directory` is non-optional** whenever one
  system user queries a repo owned by another. Without
  `/etc/gitconfig` entries, alive-svc's `git rev-parse` over the
  dotfiels tree just silently errors and the UI shows "unknown".
- **Secret Manager config is pull-based, not watch-based.**
  Updating the `attlas-server-config` secret in Secret Manager
  does NOT automatically refresh `/var/lib/alive-server/.attlas-
  server-config.json` — base-setup writes it once and the service
  reads from local disk. Needs either a base-setup re-run, a
  manual `gcloud secrets versions access | tee ...`, or a config-
  reload hook on alive-server. For today it was a manual tee.
- **The backend timeout helper (`runCmdCtx`) is worth it.** Every
  subprocess that touches systemctl, git, whois, openclaw or the
  Anthropic API is now bounded by a hard deadline, so a single
  wedged dependency can't stall `/api/status`.

## Post-plan work

Everything above was the locked `details-plan.md`. After the plan was
clean and shipped, the session kept going and tackled a few things
that the plan didn't cover but were adjacent enough to finish in the
same day.

### Card-based dashboard refactor
- Rebuilt the main dashboard around the same `Card`, `card-row`,
  `card-headline` and `card-grid` primitives as the openclaw detail
  page. No more "sections + yellow rules" hybrid style — the hero,
  cloud-spend card, infrastructure card, status card and services
  card all share the detail-page chrome. The dashboard and detail
  pages now look like they belong to the same product.

### Infrastructure detail page + Stop VM
- `pages/detail/Infrastructure.jsx` — new route at
  `/services/details/infrastructure`. Shows name, machine type, zone,
  external IP, running dot, plus primary `details` / **`stop vm`**
  actions. The Stop VM button double-confirms via `window.confirm()`
  with an explicit warning.
- Backend: `POST /api/vm/stop` — calls the Compute Engine
  `instances.stop` REST API directly using the metadata-server OAuth
  token. The scope is already on the VM service account
  (`cloud-platform`), so nothing new was needed on the IAM side.
  The API returns immediately with an Operation object, so the HTTP
  response completes fine before the OS actually goes down ~30s later.

### Combined cloud-spend card (GCP + Anthropic MTD)
- Full-width `CLOUD SPEND` card on the main dashboard: big
  `total_mtd_usd` headline, two-column breakdown underneath for
  `anthropic_mtd_usd` and `gcp_mtd_usd`. Errors for either side
  surface inline instead of crashing the card.
- `GET /api/cloud-spend` (`handleCloudSpend`):
  - Anthropic half reuses `fetchAnthropicSpend` pointed at
    month-start, **not** the last-30-day window the openclaw detail
    page uses, because the combined card is strict MTD.
  - GCP half is a SQL query against the BigQuery billing export
    (`SELECT SUM(cost) FROM <project>.billing_export.gcp_billing_
    export_v1_* WHERE project.id = <id> AND usage_start_time >=
    month_start`). Wildcard table name so we don't have to pin the
    billing account id.
  - Cached 5 minutes. Human-friendly error for the "export enabled
    in terraform but not yet wired in the console" state:
    "waiting for first billing export (up to 24h after enabling in
    console)".

### BigQuery billing export via Terraform
- New `infra/singlevm-setup/billing_export.tf`, already applied:
  - Enables `cloudbilling`, `cloudresourcemanager`, `bigquery`,
    `logging`, `billingbudgets` APIs.
  - Creates the `billing_export` dataset (EU, no expiry).
  - Grants the VM service account `roles/bigquery.dataViewer` on
    the dataset and `roles/bigquery.jobUser` on the project.
- **Known manual step**: Terraform can't flip the "send billing data
  to BigQuery" switch — that only lives in the Cloud Console
  (Billing → Billing export → BigQuery export → Edit → choose
  `petprojects-488115:billing_export`). Until that's done, the card
  shows "waiting for first billing export". Once done, data lands
  within ~24h and the card fills in automatically from the next
  5-min cache refresh.

### Unified VM uptime across the rename
- The uptime chart previously filtered on
  `protoPayload.resourceName`, which broke on the day the VM was
  renamed from `openclaw-vm` (Mar 22 → Apr 8 14:34) to
  `simple-zombie` (Apr 8 14:42 → now). Dropped the filter so the
  daily-uptime replay now covers both names as a single timeline.
  Apr 1–7 correctly show 24h; Apr 8 shows 20h 33m (up all morning,
  briefly down during the rename); Apr 9/10 show live data.

### Anthropic 100x bug + Last-30-days window
- The openclaw detail spend card had two independent bugs:
  1. The Anthropic `cost_report` response is already in USD cents,
     not micros — the old code divided by 100, i.e. it was showing
     a number 100x too high.
  2. The window was "this month" while the Anthropic console shows
     "last 30 days"; moving the openclaw page to a last-30-days
     window made the dashboard number match the console number
     (~$0.52 at the time of verification).
- These fixes only affect the openclaw detail page; the combined
  cloud-spend card on the main dashboard uses strict MTD on
  purpose, per the product intent.

### Status
- All 5 services (`alive-server`, `ttyd`, `code-server`,
  `openclaw-gateway`, `caddy`) plus `dotfiles-sync.timer` still
  `active`.
- Dashboard final state: unified hero + cards, Infrastructure and
  Openclaw detail links both live, Stop VM armed, dotfiles sync
  button working, domain banner current (363 days), cloud-spend card
  currently reading "waiting for first billing export" on the GCP
  half and the correct MTD on the Anthropic half — will flip to full
  numbers after the one-time console export action.

### Decision
- **Anthropic billing export**: not doing one. Anthropic doesn't
  publish a BigQuery billing export, and `cost_report` is the
  canonical source for MTD. Duplicating it via a scraping path adds
  failure modes for no benefit. The GCP side uses the BigQuery
  export only because there's no equivalent MTD API for GCP.
