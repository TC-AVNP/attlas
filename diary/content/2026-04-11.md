## Lessons

- **Per-repo git identity belongs in `install.sh`, not `~/.gitconfig`.** Global identity clobbers every other repo's intended author. Per-repo config in an install script scopes correctly and propagates across laptops via `git pull`.
- **`git mv` after an `Edit` does not auto-stage the content change.** The rename entry keeps the HEAD blob hash, so the index has the rename but still the OLD file content. Caught it when the recommit showed stale `com.pedro` content under the new `com.commonlisp` filename. Fix: explicit `git add` on the file after the content edit.
- **Cloud Logging audit events are the WRONG source for uptime.** They can only see API-initiated start/stop. Anything that stops the VM another way (guest-OS shutdown, host maintenance, preemption, crash) looks like "still running" to an event replay, which silently produces phantom 24h days. The real source for VM uptime is Cloud Monitoring's `compute.googleapis.com/instance/uptime` metric ŌĆö Google's own ground truth, survives every stop path.
- **Cloud Logging returns empty middle pages with live cursors.** Even with a simple filter that clearly has results in the window, the `entries:list` API can return an empty first page (or three empty pages in a row) with a `nextPageToken` that eventually lands on the real data. Pagination is not optional. Saw first-hand while debugging the old uptime implementation: page 1 had 2 stale entries, pages 2-4 had 0 entries each, page 5 had the 37 events we actually cared about.
- **Native SVG `<title>` tooltips are unusable for anything serious.** ~2-second delay before they appear, can't contain swatches or formatted rows, and on stacked bars they only apply to one segment at a time. Replace with a mousemove-driven React div any time the chart has more than one dimension.
- **A 30-second cache on a frontend that polls every 15s is indistinguishable from no cache at all.** That was the root of the Anthropic rate-limit issue ŌĆö the openclaw detail page was firing `cost_report` calls through a cache that lived for less than half a poll interval. Any external-API handler with a cache TTL shorter than the frontend's poll interval needs to have one of them adjusted.
- **The pre-close diary checklist paid for itself a second time.** Day 3 shipped missing fields; Day 4 (both the morning and this evening rewrite) hit every required field on the first pass because the checklist was there to force it.
- **Complexity accumulates quietly.** The user was right to call out the codebase mess: `main.go` was ~600 lines when Day 3 ended and ~2000 lines by now, grown feature-by-feature with no pullbacks. Refactor discipline needs to be part of the plan, not an afterthought. That's what `refactor.md` is for.
