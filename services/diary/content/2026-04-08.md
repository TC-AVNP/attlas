Built the entire attlas infrastructure from zero in a single day. Provisioned a GCP VM with Terraform, created a React dashboard with Python backend, set up Caddy as HTTPS gateway, and installed three browser-accessible services. The Claude Code PTY login was by far the hardest problem.

## What got done

- **attlas-infra**: Provisioned GCP VM with Terraform, static IP, firewall rules, and startup script
- **attlas-dashboard**: Built React dashboard with Python backend and Caddy HTTPS gateway
- **terminal**: Installed ttyd as a browser-accessible cloud terminal
- **code-server**: Installed cloud VS Code behind Caddy with subpath routing
- **openclaw**: Integrated OpenClaw Control UI behind Caddy cookie auth

## Timeline

### Infrastructure setup (~65min, 12:55 – 14:00)

- Fixed broken GCP VM networking after a terraform apply
- Created reproducibility plan: fresh VM that provisions itself

- Wrote Terraform config: VM, static IP, firewall rules, startup script

### Dashboard and base setup (~90min, 14:00 – 15:30)

- Built React dashboard app with VM info, service status, Claude login
- Python HTTP server backend on localhost:3000
- Caddy gateway with sslip.io domain and auto-HTTPS
- Service installer menu with auto-discovery
- Installed ttyd (cloud terminal) and code-server (cloud VS Code)

### Claude Code PTY login (~120min, 15:03 – 17:04)

- Full interactive TUI wizard automation via PTY (theme picker, login method, URL display, code input, security notice, trust prompt)
- Detecting screens by text triggers, pressing Enter with delays
- URL extraction was the biggest pain point (~1h alone): URLs wrap at terminal column width, ANSI escapes interleave with URL text, next prompt text gets concatenated
- Went through ~8 iterations of regex fixes for URL extraction
- Code submission: type characters one-by-one into PTY, handle post-login prompts

### Caddy routing battles (~97min, 17:05 – 18:42)

- code-server subpath routing: fought with handle vs handle_path, port changes, subpath configs
- Triple password prompt bug: basic auth applied at multiple Caddy levels
- Evolved from basic_auth to cookie-based forward_auth for single login
- Login page CSS broken: Python str.format() eating CSS curly braces

### OpenClaw integration (~72min, 18:44 – 19:56)

- Exposed OpenClaw Control UI at /openclaw/ via Caddy
- Auth headache: tried gateway token injection, trusted-proxy mode, header forwarding
- Settled on auth:none since Caddy cookie auth already protects all upstream services
- Had to set allowedOrigins for the Control UI to accept the sslip.io domain

### Wrap-up (~13min, 20:04 – 20:17)

- Wrote TODO with full summary and remaining work items
- Identified issues: single-threaded server, hardcoded secrets, etc.

## Lessons

- Claude Code's login is a full interactive TUI — no CLI flag for non-interactive auth.
- Terminal URLs wrap at column width and concatenate with surrounding text after ANSI stripping.
- code-server subpath routing requires handle_path, not handle.
- OpenClaw behind a reverse proxy is simplest with auth:none plus upstream cookie auth.
- Python's str.format() eats curly braces in CSS — use %% escaping or template strings.
