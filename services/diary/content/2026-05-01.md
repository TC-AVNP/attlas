## Lessons

- **Collision checks are critical.** Accidentally created a duplicate "export as markdown" feature (#125) that already existed as #35. This directly motivated the mandatory collision-check rule in /god — always `get_project` before `add_feature`.
- **Taxonomy should emerge from use, not be designed upfront.** Started with 4 lifecycle stages (New/Ongoing/Usable/Archived), simplified to 3 (Idea/Live/Completed) after realizing Ongoing and Usable were the same thing. Better to start simple and split later when the distinction proves necessary.
- **The MCP server IS the handoff.** The old BRIEF.md approach for handing context between planning and implementation sessions was obsolete — `get_project` via MCP gives any agent the full picture live. No static files needed.
- **Removing code is the best feature.** Deleting the canvas and list views removed 1726 lines and nearly halved the frontend bundle. The kanban replaced both with 190 lines.
- **Write the guide, then build, then audit the guide.** The "How to create" checklist started at 10 steps. After actually building knowledge.attlas.uk following it, found 5 major gaps (port allocation, Go patterns, design system, alive-server rebuild path, CLAUDE.md). Expanded to 12 steps. The guide was only solid after the gap analysis — writing it before building gave structure, building exposed what was missing.
- **alive-server binary runs from repo path, not /usr/local/bin.** The systemd unit ExecStart points to `services/alive-server/attlas-server`, not the standard binary location. Deploying to the wrong path means the old binary keeps running. This was important enough to call out explicitly in the knowledge base.
- **Kill stale test processes before deploying.** The knowledge service failed to start because a test instance was still holding port 7694. Always `fuser -k` before `systemctl restart` when testing on the same port.
- **Dual-view (human/LLM) should be day-one.** Adding it after the fact required a migration and touching every query. If the schema had both fields from the start, the content could have been written correctly the first time instead of backfilling.
