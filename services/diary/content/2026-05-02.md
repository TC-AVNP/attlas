## Timeline

### Mobile scroll button fix (~4min, 10:03 – 10:07)

- Identified that `#mk-scroll` was inside `#mk` (the keyboard overlay, hidden by default) while desktop had a separate `#mk-scroll-desktop` in `#mk-desktop-btns`
- Added `#mk-scroll-mobile` button to `#mk-toggles` — the floating button group always visible on touch devices
- Wired it to the existing `_mkToggleScroll()` function with touchstart + click handlers
- Re-ran `install.sh` to rebuild the custom ttyd index with the updated fragment
- Verified the button appears in the deployed HTML

### Planning handoff feature (~5min, 09:37 – 09:42)

- Used /god to plan feature #158: project action bar with Handoff and Work on it buttons
- Ran collision check against existing features — found overlaps with #35 (done, lost in UI rebuild) and #84 (Start building, already shipped)
- Resolved: Handoff restores lost export, Work on it reuses existing terminal launcher

### Building the handoff feature (~25min, 09:42 – 10:08)

- Read codebase: Kanban.tsx, ProjectDetail.tsx, api client, types
- First iteration: two separate buttons (Handoff + Work on it) — user wanted single Actions dropdown
- Second iteration: single Actions dropdown menu — user wanted two buttons, not a dropdown
- Final iteration: two buttons in header, Handoff opens a modal with project description, "Select details" expandable section with feature checkboxes, Download button
- Built markdown generation: project name, problem, description, features grouped by status
- Deployed three times iterating on the UX based on user feedback

## Project summaries

### terminal

Fixed mobile scroll button visibility. The scroll toggle (tmux copy-mode) was only accessible when the custom keyboard was open. Added a dedicated scroll button to the `#mk-toggles` floating group so it's always reachable on mobile, matching the desktop experience where it's in `#mk-desktop-btns`.
**Lines:** 1-122

### petboard

Added feature #158 — project action bar on the detail page header. Two buttons: "Handoff" opens a modal explaining the download with an expandable feature selector (all pre-ticked), downloads a markdown file with full project context. "Work on it" is a greyed-out placeholder. Involved three iterations to get the UX right based on user feedback.
**Lines:** 3-210

## Lessons

- The terminal's mobile UI has two layers: `#mk-toggles` (always visible floating buttons) and `#mk` (the full keyboard overlay, toggled on/off). Features that should always be accessible must go in the toggles, not inside the keyboard.
- Listen more carefully to the user's UI description before building. "A button with two options" means a single trigger with choices inside, not two buttons or a dropdown. Three iterations could have been one.
- When views get deleted (canvas, list view), audit for orphaned features that depended on UI elements in those views. The export from #35 was silently lost when the kanban replaced the list view.
